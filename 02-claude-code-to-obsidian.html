Build 02 // obsidian plugin

Every Claude Code session,
filed in your vault as you go.

Claude Code already writes a full record of every session to disk as plain JSON Lines. No login, no scraping, no gray area. This plugin reads those files and turns each session into a clean Markdown note, refreshed on a timer or on demand.

~/.claude/projects → read .jsonl → parse turns → to Markdown → Vault API → your notes
Why this one is the easy build

Unlike claude.ai chats, Claude Code stores everything locally and append-only. Each working directory you have used maps to a folder of .jsonl files, one file per session, every line a complete JSON message. It is greppable, backup-able, and yours. An Obsidian plugin runs in Electron, so it has full Node fs access to read them. Nothing here reaches across a network.

0

Know where the sessions live

~/.claude/projects/<encoded-cwd>/<sessionId>.jsonl

Claude Code keeps sessions under ~/.claude/projects/. Each subfolder is one project directory, encoded by taking the absolute path and replacing / and . with -. Inside are the session files.

~/.claude/projects/
-Users-matt-code-quickreads/
  a1b2c3d4-....jsonl      # one session
  e5f6a7b8-....jsonl      # another session
-Users-matt-code-birchfeed/
  ...

Because the folder name decodes back to the original path, you get the project name for free. -Users-matt-code-quickreads tells you the session belongs to quickreads, which becomes the note's folder or a tag.

Scaffold the plugin

manifest, entry point, build

An Obsidian plugin is a tiny TypeScript project: a manifest.json, a main.ts that exports a Plugin subclass, and an esbuild step that bundles to main.js. Start from the official sample plugin template and strip it down.

manifest.json
{
  "id": "claude-code-sync",
  "name": "Claude Code Sync",
  "version": "0.1.0",
  "minAppVersion": "1.5.0",
  "description": "Mirror Claude Code sessions into your vault as Markdown.",
  "isDesktopOnly": true   // needs Node fs, so desktop only
}
main.ts — skeleton
import { Plugin } from "obsidian";

export default class ClaudeCodeSync extends Plugin {
  async onload() {
    // a ribbon button for manual sync
    this.addRibbonIcon("sync", "Sync Claude Code", () => this.syncAll());

    // a command palette entry
    this.addCommand({ id: "sync-now", name: "Sync now",
      callback: () => this.syncAll() });

    // run on load, then on an interval (stage 5)
    await this.syncAll();
  }

  async syncAll() { /* stages 2-4 */ }
}
2

Read and parse the JSON Lines

one line, one message

Walk the project folders, read each .jsonl, and parse it line by line. Every line is an independent JSON object, so a single malformed line should be skipped, not fatal.

reading a session file
import * as fs from "fs";
import * as os from "os";
import * as path from "path";

function readSession(file: string) {
  const lines = fs.readFileSync(file, "utf8").split("\n");
  return lines
    .filter(l => l.trim())
    .map(l => { try { return JSON.parse(l); } catch { return null; } })
    .filter(Boolean);
}

The event shape is not officially documented and can shift between versions, so inspect a real file before trusting field names. A user turn typically looks like this:

a representative line (trimmed)
{
  "type": "user",
  "uuid": "...", "parentUuid": "...",
  "timestamp": "2026-07-06T18:04:11Z",
  "sessionId": "a1b2c3d4-...",
  "cwd": "/Users/matt/code/quickreads",
  "message": { "role": "user", "content": "add a retry to the TTS call" }
}
Content is sometimes a string, sometimes blocks

Assistant turns often carry an array of content blocks: text, tool_use, and tool_result. Handle both. Flatten text blocks into the body, and decide whether tool calls become collapsed callouts or get dropped. Lines with type: "summary" are Claude Code's own recap and make a great note title.

Shape it into a note

frontmatter plus turns

Order the events (file order is usually fine, or walk parentUuid for the true thread), then render each turn under a speaker heading. Lead with YAML frontmatter so the note is queryable by Dataview later.

the Markdown you generate
---
session: a1b2c3d4
project: quickreads
date: 2026-07-06
source: claude-code
---

# add a retry to the TTS call

## You · 18:04
add a retry to the TTS call

## Claude · 18:04
Wrapped the Async TTS request in a backoff loop...

> [!tool]- ran: edit src/tts.ts
> +  await retry(() => ttsClient.speak(text))

Folding tool calls into Obsidian's collapsible [!tool]- callouts keeps the transcript readable while preserving what actually happened. Strip Claude Code's internal noise (system reminders, empty tool acks) so the note reads like a conversation, not a log dump.

4

Write through the Vault API

not raw fs, so Obsidian indexes it

Read the source files with Node fs, but write into the vault with Obsidian's own API. That way the app indexes the note, updates links, and shows it live. Create the note if it is new, modify it if it already exists.

upsert into the vault
async writeNote(folder: string, name: string, md: string) {
  const dir = this.settings.targetFolder + "/" + folder;
  if (!this.app.vault.getAbstractFileByPath(dir))
    await this.app.vault.createFolder(dir);

  const filePath = `${dir}/${name}.md`;
  const existing = this.app.vault.getAbstractFileByPath(filePath);

  if (existing) await this.app.vault.modify(existing as TFile, md);
  else        await this.app.vault.create(filePath, md);
}
5

Keep it in sync, incrementally

interval plus a high-water mark

Two pieces make it a real sync. A schedule, and a memory of what you already handled. Obsidian's registerInterval cleans itself up on unload, so use it rather than a bare setInterval.

schedule + skip unchanged files
// in onload(), after the first sync
this.registerInterval(
  window.setInterval(() => this.syncAll(), this.settings.intervalMs)
);

// inside syncAll(): only touch files that grew
const stat = fs.statSync(file);
const seen = this.state[file]?.mtimeMs ?? 0;
if (stat.mtimeMs <= seen) return;        // unchanged, skip
// ...parse, write...
this.state[file] = { mtimeMs: stat.mtimeMs };
await this.saveData(this.state);      // persists across restarts

Comparing file mtime against a stored value means a run over an idle project does almost nothing. For very active sessions you can go further and track the last line count, appending only new turns instead of rewriting the whole note.

Give it a settings tab

the three knobs people will want

Add a PluginSettingTab so nobody has to edit code. Keep it to what actually varies.

  • Source path. Default ~/.claude/projects, overridable for non-standard setups.
  • Target folder. Where notes land in the vault, for example Claude Code.
  • Interval. How often to sync, with an option for manual-only.
  • Tool calls. A toggle: fold them into callouts, or leave them out.

Ship it

install, then edge cases

For your own use, drop main.js, manifest.json, and styles.css into <vault>/.obsidian/plugins/claude-code-sync/ and enable it in settings. To let others try it before it is in the community list, point them at BRAT, which installs plugins straight from a GitHub repo. To go fully public, submit to the community plugins directory.

The edges that will bite

Files change mid-read. A session can be written while you parse it, so read defensively and let the next interval catch up. Big sessions. Long transcripts make large notes, so consider a size cap or splitting by day. Desktop only. There is no fs on mobile, which is why the manifest sets isDesktopOnly. Format drift. The .jsonl schema is undocumented, so guard every field access and fail soft when a version changes something.
